# 《天穹断刃》4.0 设计基线
## 基于《Hades》的结构拆解与网页机甲肉鸽重构方案

> 文档目的：停止“边写代码边补功能”的原型式迭代，先明确核心循环、系统分层、内容生产规则和验收标准，再进入 4.0 开发。

---

## 一、先纠正此前方向

此前版本的问题不只是模型、特效或单个 Bug，而是缺乏完整的产品骨架：

- 战斗房间之间没有明确的节奏差异。
- 强化只是随机数值卡片，没有围绕武器动作形成构筑。
- 死亡、返航、角色关系和局外成长没有形成第二条循环。
- 场景、敌人和奖励没有被同一个“关卡导演系统”统筹。
- 动作、硬点、伤害判定和特效各自计算，造成枪口与弹道、挥砍与伤害区域错位。
- 代码以单文件快速堆叠为主，缺少可测试的模块边界。

因此 4.0 不再以“增加更多功能”为目标，而是重做底层结构。

---

## 二、《Hades》的核心设计不是“随机房间 + 三选一”

《Hades》官方将其定义为快节奏动作、角色驱动叙事与重复挑战结合的 rogue-like dungeon crawler。每次尝试都重新组合能力、挑战和故事事件；死亡后玩家携带知识与部分永久成长进入下一次尝试。

它真正成立依靠五条互相咬合的系统链：

1. **瞬时战斗循环**：移动、攻击、特殊攻击、投射/技能、闪避，全部具有明确用途和取消关系。
2. **单局构筑循环**：武器基础动作固定，祝福和锤子改变动作规则，而非只增加数值。
3. **房间决策循环**：奖励在进入前可预判，玩家主动选择下一步构筑方向。
4. **局外成长循环**：失败后仍能获得永久成长、武器解锁、资源、关系和新对话。
5. **叙事反馈循环**：角色记得玩家做过什么、死于谁、到过哪里，使失败本身也产生内容。

这五条循环同时工作，才会让重复闯关不显得重复。

---

## 三、Hades 的三层循环

### 3.1 秒级循环：战斗动作

一个成熟动作肉鸽的基本要求：每个输入都必须形成完整链条。

```text
输入 → 动作状态 → 动画/姿态 → 武器硬点 → 命中判定 → 受击反馈 → 特效/音效/震动
```

任何一环脱离都会出现此前版本的问题：

- 枪在右手，光束却从身体中心生成。
- 右键有伤害弧线，但机体没有挥刀动作。
- 敌人扣血，却没有停顿、偏转、后仰和装甲破坏反馈。

Hades 的战斗重点不是动作数量很多，而是少量基础动作之间具有明确关系：

- 普通攻击：稳定输出和连段主轴。
- 特殊攻击：武器差异最大的第二主轴。
- 投射/法术：补充控制、标记、爆发或远程能力。
- 闪避：移动、防御、取消和进攻位移共用。
- 终极能力：由战斗积累，在关键节点释放。

### 3.2 分钟级循环：房间与构筑

```text
进入房间
→ 识别敌人与地形
→ 清场
→ 获取明确奖励
→ 比较下一房间门牌
→ 选择构筑方向或资源方向
→ 进入下一房间
```

关键原则：

- 奖励选择要发生在进入房间前，而非清场后完全随机。
- 不同房间承担不同功能：战斗、精英、商店、恢复、事件、剧情、Boss。
- 一个区域内部需要形成“普通战 → 强化 → 风险房 → 补给 → Boss”的节奏曲线。
- 随机化负责改变路径组合，不负责替代关卡设计。

### 3.3 局级循环：失败、返航与世界推进

```text
一次出击结束
→ 回到基地
→ 结算资源
→ NPC 根据本次经历触发对话
→ 解锁永久能力/装备/剧情
→ 重新选择机体与武器
→ 下一次出击
```

这里是 Hades 与普通房间肉鸽差异最大的部分：

- 死亡不是空白重置。
- 局外空间承担剧情、教学、目标提示和情绪缓冲。
- 对话系统要读取本次出击产生的标签，而不是固定顺序播放。
- 永久成长同时服务于降低门槛和提供长期目标。

## 四、Hades 的构筑为什么有效

### 4.1 构筑建立在“动作槽位”上

强化不会全部混在一个数值池，而是优先绑定到基础动作：

- 普通攻击槽
- 特殊攻击槽
- 投射技能槽
- 闪避槽
- 终极槽
- 被动效果槽

这样玩家能快速理解：这张卡到底改变了哪个按钮。

### 4.2 强化分为三类

1. **数值强化**：伤害、速度、暴击、范围。
2. **规则强化**：弹射、回旋、滞留、召回、标记、爆炸。
3. **联动强化**：满足两个体系后出现的组合能力。

真正支撑重复游玩的，是后两类。

### 4.3 单局与局外要分开

单局获得：

- 临时武器模块
- 技能协议
- 消耗品
- 临时货币

局外保留：

- 新机体
- 新武器
- 武器形态
- 基础属性树
- NPC 关系
- 图鉴和挑战等级

如果所有东西都在单局内，死亡会显得浪费；如果永久成长太强，单局构筑会失去意义。

## 五、转译成机甲题材后的产品结构

## 5.1 产品定位

**一句话：**

> 固定斜俯视角的高速机甲动作肉鸽，玩家驾驶超级系原创机体，在一次次战役中重构武装与机体协议，并通过返航基地推进角色关系和世界事件。

### 设计支柱

1. **看得懂**：敌我轮廓、攻击来源、危险区和奖励必须一眼可读。
2. **动作对齐**：模型姿态、武器硬点、判定和特效由同一套数据驱动。
3. **构筑改变动作**：优先改变射击方式、军刀动作、导弹逻辑和推进规则。
4. **失败也推进**：每次返航都有资源、情报、对话或解锁。
5. **短局完整**：网页手机版目标单次完整战役 20–30 分钟，单房间 30–75 秒。

---

## 5.2 双循环结构

### 单局循环

```text
选择机体与主武器
→ 进入区域
→ 在两个可见奖励房间中选择
→ 战斗/事件/商店/维修
→ 精英战获取高级模块
→ 区域 Boss
→ 进入下一环境
→ 最终 Boss 或战败
```

### 局外循环

```text
返回母舰
→ 战役报告
→ NPC 条件对话
→ 使用战役资源解锁机体/武器/协议
→ 查看图鉴和挑战任务
→ 调整初始配置
→ 再次出击
```

## 六、4.0 的内容架构

### 6.1 机体

首发纵切只做三台，但必须真正差异化：

#### 断刃·先锋型
- 主武器：光束步枪
- 特殊攻击：回旋军刀斩
- 推进：短冷却高速闪避
- 构筑关键词：连射、折射、近远切换

#### 断岳·重装型
- 主武器：爆裂加农炮
- 特殊攻击：盾击/蓄力重斩
- 推进：短距离装甲冲撞
- 构筑关键词：爆炸、护盾、反击、过热

#### 星翼·高机动型
- 主武器：双联光束枪
- 特殊攻击：浮游刃突袭
- 推进：连续位移
- 构筑关键词：锁定、多目标、残影、无人武器

### 6.2 动作槽位

每台机体统一拥有：

- `Primary`：主射击
- `Secondary`：近战或特殊武器
- `Dash`：推进
- `Ordnance`：导弹/浮游炮/重火力
- `Overdrive`：必杀或超限模式

强化必须声明作用槽位，例如：

```json
{
  "id": "beam_ricochet_01",
  "slot": "Primary",
  "tags": ["beam", "ricochet"],
  "requires": ["beam"],
  "changes": {
    "bounceCount": 1,
    "damageMultiplier": 0.82
  }
}
```

### 6.3 区域

4.0 纵切先做一个真正完整的区域，而不是十二个换皮区块。

#### 区域：轨道残骸带

房间池：

- 4 个基础战斗模板
- 2 个远程压制模板
- 2 个近战围攻模板
- 2 个环境危险模板
- 2 个精英模板
- 1 个补给房
- 1 个商店
- 2 个事件房
- 1 个中 Boss
- 1 个区域 Boss

每个模板要有不同空间结构、出生节奏和敌人组合。

### 6.4 房间奖励

玩家在门口看到明确图标：

- 武器协议
- 防御协议
- 推进协议
- 导弹协议
- 战役货币
- 永久资源
- 维修
- NPC/事件

选择房间就是选择未来构筑，不是选择一块随机地图。

## 八、必须重做的技术系统

### 8.1 Hardpoint Rig：武器硬点系统

模型渲染和攻击生成必须共享骨骼/节点矩阵。

```text
Mech Root
├── Torso
│   ├── Right Shoulder
│   │   └── Right Forearm
│   │       └── Rifle
│   │           └── Muzzle
│   └── Left Shoulder
│       └── Saber Hilt
│           └── Saber Tip
└── Backpack
    ├── Missile L
    └── Missile R
```

每帧先计算节点世界矩阵，再：

- 用 `Muzzle.worldPosition` 生成光束。
- 用 `SaberHilt` 与 `SaberTip` 生成刀光带和碰撞扫掠。
- 用 `MissileL/R` 生成导弹。
- 用 `Thruster` 生成尾焰。

禁止再使用“玩家坐标 + 人工偏移值”发射武器。

### 8.2 动作状态机

```text
Idle
Move
DashStartup → DashActive → DashRecovery
PrimaryFire
SecondaryStartup → SecondaryActive → SecondaryRecovery
HitStun
Knockdown
Overdrive
Dead
```

每个状态声明：

- 持续时间
- 可取消窗口
- 移动倍率
- 上半身姿态
- 武器硬点动画
- 判定激活帧
- 特效触发帧
- 音效触发帧

### 8.3 敌人 AI 状态机

```text
Spawn
Acquire
Approach / Strafe / Retreat
Telegraph
Attack
Recover
HitReact
Stagger
Enraged
Dead
```

受击必须根据：

- 攻击方向
- 伤害强度
- 当前霸体值
- 攻击类型

计算后仰、侧倾、位移、停顿和硬直。

### 8.4 VFX 分层

禁止用大量随机小颗粒替代主体特效。

每个攻击至少包含：

- 发射层：枪口形状、光源、冲击环
- 弹道层：核心、外辉光、拖尾
- 命中层：定向闪光、冲击形状、装甲碎片
- 环境层：地面光、烟尘、残留痕迹

粒子仅作为辅层。

## 九、叙事系统基线

4.0 先搭系统，不需要马上写几千条对白。

每次出击记录标签：

```json
{
  "reachedRegion": 2,
  "killedBoss": "warden_alpha",
  "deathCause": "sniper_beam",
  "weaponUsed": "beam_rifle",
  "lowHpVictory": true,
  "metEvent": "wrecked_pilot"
}
```

返航后 `NarrativeDirector` 根据：

- 前置条件
- 优先级
- 是否播放过
- 冷却次数
- 本次战役标签

选择对话。这样角色才会“记得”玩家经历。

## 十、移动版不是桌面版加虚拟按钮

iOS 版本必须单独设计：

- 左摇杆移动。
- 右摇杆瞄准，超过阈值自动持续射击。
- 军刀、推进、导弹、必杀独立按钮。
- 自动吸附辅助瞄准，但不替玩家自动选择所有目标。
- 动态降低同时敌人数，提升单个敌人的可读性。
- 镜头稍高、危险区更粗、字体和奖励卡更大。
- 触控按钮响应必须在按下瞬间进入输入缓冲，不能等抬手。

## 十一、4.0 纵切版本的范围

不再虚报“Steam 完整版”。下一阶段只承诺一个高质量纵切：

- 1 个母舰基地界面
- 3 台可选机体
- 1 个完整区域
- 12–16 个房间组成一次战役
- 6 类普通敌人
- 2 类精英
- 1 个中 Boss
- 1 个三阶段区域 Boss
- 5 个动作槽位
- 30 个基础模块
- 8 个稀有规则模块
- 6 个双体系联动模块
- 2 个事件角色
- 返航结算和条件对话
- 键鼠、手柄和 iOS 触控输入

质量优先级：

```text
战斗动作对齐
> 操作手感
> 敌人和房间节奏
> 构筑差异
> 受击反馈
> 场景可读性
> 内容数量
```

## 十二、验收标准

### 战斗

- 所有弹道从实际武器硬点生成。
- 军刀视觉轨迹、碰撞扫掠和机体动作来自同一节点序列。
- 任何命中都能观察到敌机方向性受击反馈。
- 玩家可在 3 秒内理解敌方攻击预警。
- 闪避具有明确启动、无敌和恢复窗口。

### 构筑

- 每个动作槽位至少有 4 条不同构筑路线。
- 至少一半高级强化改变动作规则，而非单纯增加伤害。
- 同一机体两次出击可以形成明显不同的操作体验。

### 房间

- 连续三个战斗房不能使用相同布局和敌人节奏。
- 玩家进入房间前知道主要奖励类型。
- 普通房间目标时长 30–60 秒，精英 60–90 秒。

### 移动端

- iPhone 横屏无页面滚动、缩放和系统手势误触。
- 双摇杆与技能按钮支持至少三指并发。
- 触控按下到动作状态切换低于一帧逻辑延迟。
- 中场菜单切换后触控状态完全清空，不出现按钮卡住。

### 稳定性

- 30 分钟自动战斗压力测试无异常。
- 导弹目标死亡、房间清场、暂停、切后台、旋转屏幕均有回归测试。
- 每次构建自动执行语法、系统和完整流程测试。
- 错误面板只用于开发版本，发行构建使用错误日志与安全回退。

## 十四、调研结论

我们要学习 Hades 的不是表面形式，而是它的系统闭环：

- 每个战斗按钮有清晰身份。
- 强化改变动作规则。
- 房间选择等于构筑选择。
- 死亡触发局外成长和叙事。
- 模块化结构支持持续扩展和玩家反馈。

因此 GitHub 新仓库应该从这份架构开始，而不是先把 v3.2 单文件上传后继续堆代码。
