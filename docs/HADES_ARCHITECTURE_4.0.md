# 《天穹断刃》4.0 历史架构基线与 4.2 实现状态

> 这份文档最初用于指导 4.0 从单文件原型转向模块化动作肉鸽。历史设计推导继续保留，但其中部分具体实现已经被 4.2 替代。
>
> 当前事实来源优先级：`docs/MECHA_MARCO_DESIGN_4.2.md` → 当前代码与测试 → 本文的 4.2 状态注记 → 本文的 4.0 历史方案。

## 0. 版本状态总览

当前正式版本：`4.2.1-mobile-frontline-polish`

当前已经发生的关键变化：

- 战斗镜头已经从早期设想的固定斜俯视改为纯俯视 `Top-down`
- 正式战斗机体以俯视上半身轮廓为主，腿部在战斗显示中隐藏
- 单局主流程已经从“独立房间 → 路线门牌 → 独立房间”改为 12 段连续战线
- 普通段落清场后保持玩家控制，通过物理闸门与连接航道进入下一段
- 路线页面只保留为关键分岔，不再是每关必经页面
- 4.0 的模块化、Hardpoint Rig、动作槽位、构筑、基地循环和条件叙事方向继续有效
- 4.2 新增 MA-00 人格恢复、双层机体档案、异常战役报告和区域级连续战役导演

以下章节会明确标注“仍有效”“已实现”“部分过期”或“已被替代”。

## 一、历史问题诊断（仍有效）

4.0 启动前的问题并不局限于模型、特效或单个 Bug：

- 战斗段落缺少节奏差异
- 强化停留在随机数值卡片，缺少动作规则变化
- 死亡、返航、角色关系和局外成长没有形成第二条循环
- 场景、敌人和奖励没有被统一导演系统统筹
- 动作、硬点、判定和特效各自计算，造成枪口与弹道、挥砍与伤害错位
- 单文件快速堆叠，缺少可测试的模块边界

这些判断仍然成立，也是当前继续迭代时的检查标准。

---

## 二、从 Hades 学习的系统闭环（仍有效）

项目借鉴的是系统闭环，不是表面房间形式：

1. 秒级战斗循环：移动、射击、军刀、挂载、推进、超限之间有明确职责和取消关系
2. 单局构筑循环：强化优先改变动作规则，而非只叠百分比
3. 路径与风险循环：玩家在关键节点选择奖励、风险和资源方向
4. 局外成长循环：失败后仍保留资源、档案、记录、关系和信息
5. 叙事反馈循环：角色根据玩家真实战斗行为回应

### 秒级链路

```text
输入
→ 动作状态
→ 机体姿态
→ 武器硬点
→ 命中判定
→ 受击反馈
→ 特效 / 音效 / 震动
```

任何一环脱离都会重新出现旧问题：枪在右手但弹道从身体中心出现、刀光和伤害区错位、敌人扣血却没有可读反馈。

### 单局构筑原则

构筑继续绑定动作槽位：

- `Primary`：主射击
- `Secondary`：军刀或特殊攻击
- `Dash`：推进
- `Ordnance`：导弹、浮游武器或重火力
- `Overdrive`：超限 / 必杀

强化分为：

- 数值强化
- 规则强化
- 联动强化

真正支撑重复游玩的仍然是规则强化和联动强化。

## 三、4.0 历史房间循环（部分过期）

4.0 原始方案是：

```text
进入独立房间
→ 识别敌人与地形
→ 清场
→ 获取奖励
→ 比较下一房间门牌
→ 选择下一房间
```

这套结构在 4.0 / 4.1 原型中确实被实现过，也验证了奖励预判、商店、维修和事件节点。

### 过期部分

以下内容不再描述当前正式主循环：

- 每个战斗节点结束后返回全屏路线页面
- 每一关都重新载入一个独立小房间
- “门牌路线选择”作为每关固定节奏
- 以 10 个独立节点构成轨道墓场

### 4.2 当前替代方案

```text
进入轨道墓场
→ 连续完成当前战斗段
→ 闸门开启
→ 保持控制，驾驶机体穿过连接航道
→ 进入下一段战场
→ 只在关键分岔、奖励、商店、维修或事件时暂停
→ 完成第 12 段 Boss
```

4.2 仍保留“选择等于构筑方向”的原则，但选择被压缩到有意义的关键节点，避免每关都被 UI 打断。

## 四、产品定位修订

### 4.0 历史表述（已过期）

早期一句话定位曾写为“固定斜俯视角的高速机甲动作肉鸽”。

### 4.2 当前表述

> 纯俯视 Top-down 的高速机甲动作肉鸽。玩家驾驶以俯视上半身轮廓呈现的原创机体，在 12 段连续战线中重构武装，并通过返航基地推进 MA-00 人格恢复、角色关系和世界事件。

### 当前设计支柱

1. 看得懂：敌我轮廓、攻击来源、危险区、目标和奖励一眼可读
2. 动作对齐：姿态、硬点、判定与特效共享数据
3. 构筑改变动作：射击、军刀、推进、挂载和超限产生规则差异
4. 连续推进：普通战斗段之间不反复返回路线页
5. 失败也推进：资源、档案、人格恢复、对白和记录继续积累
6. 短局完整：第一章形成可完整通关的连续战役

## 五、当前双循环结构

### 5.1 单局循环（4.2 当前）

```text
选择机体 / 涂装 / 操作设置
→ 进入轨道墓场
→ 12 段连续战线
→ 关键分岔、奖励、商店、维修和事件
→ 守墓者·阿尔法
→ 战役报告
```

### 5.2 局外循环（仍有效，部分实现）

```text
返回母舰
→ 结算资源和战役标签
→ 条件对白
→ 更新人格恢复、指挥权限和档案
→ 选择机体、涂装与设置
→ 再次出击
```

当前已经实现战役记录、条件对白、人格恢复、机体档案和永久资源接口；完整科技树、关系系统和长期挑战仍未完成。

## 六、三台机体基线

### 断刃·先锋型

- 定位：近远切换的泛用王牌机
- 公开象征：职业军人、荣誉、秩序和仍有底线的文浩马
- 构筑关键词：连射、折射、近远切换、军刀联动
- 当前视觉：锐利肩甲、步枪、军刀、均衡背包

### 断岳·重装型

- 定位：攻城、爆裂和护盾反击
- 公开象征：重装突破机
- 隐藏象征：文浩马从王牌转向统治者的城市级攻城工具
- 构筑关键词：爆炸、护盾、反击、重火力
- 当前视觉：宽肩厚胸、重型武器、双反应炉背包

### 星翼·高机动型

- 定位：多目标、高机动和浮游武器
- 公开象征：实验型截击机
- 隐藏象征：统治后期高效率处决与自动索敌
- 构筑关键词：锁定、多目标、残影、浮游单元
- 当前视觉：窄躯干、展开翼板、多推进节点和浮游挂载

三台机体目前已经具备属性、动作倾向、涂装和可见挂载差异；完全独立的专属资源机制仍是后续重点。

## 七、轨道墓场区域结构（4.2 当前）

当前第一区域由 12 段连续战线组成：

1. 封锁航道
2. 残骸雨带
3. 断裂船坞
4. 身份识别阵列
5. 封锁维修栈桥
6. 英雄纪念舰列
7. 封印输能带
8. 监察官追猎
9. 墓碑岔道
10. 核心外环
11. 守墓者前庭
12. 轨道墓场核心

### 关键分岔

- 断裂船坞：军械库航线 / 数据侧路
- 墓碑岔道：维护线 / 旧档案库

分岔会改变奖励、维修、精英增援、识别次数和档案恢复，不只是替换文本。

### 仍未完成

多数段落目前仍以清敌为最终完成条件。后续需要把目标文本继续落实成真实玩法：

- 摧毁识别塔
- 摧毁封印输能设施
- 限时守点
- 跨段追击监察官
- 核心外环动态闸门
- Boss 与封印装置联动

## 八、代码架构现状

4.0 提出的模块化方向已经落地。当前主要结构包括：

```text
v4/src/
  core/       循环、事件总线、数学与状态机
  render/     Canvas 场景、WebGL 机体、纯俯视镜头与区域渲染
  combat/     Ability、Hardpoint、命中、移动手感与构筑运行时
  actors/     玩家与敌人
  run/        奖励、协议、连续战役导演
  meta/       档案、存档、叙事与人格恢复
  data/       机体、模块、敌人、事件与轨道墓场数据
  ui/         基地、HUD、构筑、涂装与战役通讯
  tests/      系统、视觉接线、连续战役和文档一致性测试
```

### 当前连续战役关键层

- `data/regionOrbitalGraveyard42.js`：12 段数据、分岔、通讯和事件
- `run/continuousCampaign42.js`：连续推进、奖励、事件、报告和战役状态
- `run/continuousCampaignPolish42.js`：连续战役收尾修正
- `render/continuousCampaign42.js`：区域环境和闸门表现
- `ui/campaignUI42.js`：进度、通讯、分岔和异常报告
- `meta/restoration42.js`：人格恢复、匹配率和指挥权限

旧 `roomGraph.js` 与门牌式房间图仍保留为历史 / 兼容系统，但不再主导 4.2 轨道墓场正式流程。

## 九、Hardpoint Rig 与动作系统（仍有效，已实现）

模型和攻击继续共享硬点链：

```text
Mech Root
├── Torso
│   ├── Right Shoulder → Right Forearm → Rifle → Muzzle
│   └── Left Shoulder → Saber Hilt → Saber Tip
└── Backpack → Missile / Thruster / Drone Nodes
```

运行原则：

- 光束从实际枪口节点生成
- 军刀视觉与判定共享刀柄 / 刀尖轨迹
- 导弹从背包挂载生成
- 推进焰从推进节点生成
- 商店或奖励修改装备后，视觉挂载和战斗能力同步刷新

禁止重新退回“玩家中心坐标 + 人工偏移”生成武器的做法。

## 十、纯俯视战斗显示基线（4.2 当前）

### 镜头

- 世界到屏幕映射使用纯俯视投影
- 不使用斜切、平台透视深度或三分之四战斗镜头
- 镜头只做有限跟随和冲刺响应，保证战场边界稳定可读

### 机体轮廓

- 战斗中优先显示头部、胸甲、肩甲、双臂、武器、背包、翼板、挂舱和推进器
- 腿部在正式战斗姿态中隐藏，避免俯视轮廓出现不合理的完整站立身体
- 基地预览仍可展示更完整机体结构
- 玩家、精英和 Boss 保留完整硬表面表现；普通敌人使用更低成本结构

### 可读性要求

- 玩家和敌方轮廓必须通过形状、朝向、武器和发光节点区分
- 危险区、锁定、Boss 血条和通讯不得被触控区遮挡
- 低饱和 UI 只负责信息，不抢夺战斗主体

## 十一、叙事系统基线（4.2 已扩展）

4.0 的 NarrativeDirector 条件对白方向继续有效。4.2 在此基础上增加：

- `WENHAO MA` / `MA-00` 身份识别
- 人格恢复分
- 指挥权限
- Boss 见面与击杀
- 机体使用次数
- 主射击、军刀和挂载击杀倾向
- 高风险路线
- 低血清场
- 投降接受 / 拆解
- 档案碎片

剧情不能只依赖死亡次数。不同玩家会以不同顺序看到恢复迹象，但主线结论保持一致。

第一章只揭示：

- 敌方认识文浩马
- 轨道墓场不是普通军事基地
- 母舰成员隐瞒身份信息
- MA-00 权限正在恢复
- 守墓者认为他不该被带回来

第一章不直接解释文浩马曾经统治世界、舰长零的本质或复归计划全部技术原理。

## 十三、验收标准修订

### 战斗

- 弹道、军刀、导弹和推进来自实际硬点
- 命中具有方向性反馈、停顿、震动和清晰特效层
- 纯俯视轮廓下可快速理解敌方攻击预警
- 推进具有明确启动、无敌和恢复窗口

### 构筑

- 高级强化优先改变动作规则
- 同一机体两次出击可形成明显不同的操作体验
- 安装模块后，战斗参数和可见装备同时变化

### 连续战线

- 12 段按固定战役节奏连续推进
- 普通段清场后不返回全屏路线页
- 闸门开放后玩家保持控制并进入下一段
- 关键分岔真实改变后续状态
- 连续三个战斗段不能完全复用相同布局、敌军节奏和环境危险

### 移动端

- iPhone 横屏无页面滚动和布局溢出
- 双摇杆与技能按钮支持多指并发
- 菜单切换后输入状态完全清空
- 触控区不遮挡玩家、Boss 与核心战场信息

### 稳定性

- ES 模块语法、Node 系统测试、WebGL Smoke 和 Pages Smoke 自动执行
- 12 段流程、奖励、商店、分岔、事件、Boss 与结算纳入真实浏览器长流程
- 文档、VERSION、页面标题、package.json 和 Service Worker 缓存版本保持一致

## 十六、历史调研结论（继续保留）

项目要学习 Hades 的仍然是系统闭环：

- 每个战斗按钮有清晰身份
- 强化改变动作规则
- 关键路径选择等于构筑和风险选择
- 死亡触发局外成长与叙事
- 模块化结构支持持续扩展和玩家反馈

变化只发生在空间组织方式：4.2 不再机械复制“每关门牌选房”，而是把这些原则重新组织成适合机甲推进感的连续战线。
